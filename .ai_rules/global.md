# AI & Developer Guide: Global Project Settings
이 모듈 및 설정은 Python 3.12+, uv, ruff, pytest 등 프로젝트 전반의 툴체인, 코딩 컨벤션, 인프라 및 에이전트 협업 가이드라인을 정의하고 준수하도록 돕는 전역 인프라 역할을 담당합니다.

## 🤖 AI Agent Guidelines
- **언어 및 런타임 제약**: 반드시 Python 3.12+ 문법과 최신 기능을 사용하며, 모든 새로운 Python 파일 상단에는 `from __future__ import annotations`를 반드시 명시해야 합니다.
- **패키지 및 종속성 관리**: 패키지 설치 및 실행은 반드시 `uv`를 사용하며, 종속성은 `pyproject.toml`을 통해서만 관리합니다. 개발 환경 내에서 임의로 `pip`를 사용해 패키지를 설치해서는 안 됩니다.
- **코드 스타일 및 린터**: 코드 생성 또는 수정 후에는 반드시 `ruff check .` 및 `ruff format .`을 실행하여 린트 오류 및 포맷 규칙을 통과해야 합니다. 기존의 Ruff 린트 규칙을 우회하기 위한 조치를 임의로 추가하지 마십시오.
- **테스트 강제**: 기능 변경 또는 버그 수정 시에는 반드시 `tests/` 폴더 내에 `test_` 접두사를 가지는 pytest 파일을 추가하거나 기존 테스트를 업데이트해야 하며, 제출 전 `uv run pytest`가 정상 통과하는지 확인해야 합니다.
- **엄격한 타입 힌팅**: 모든 public 함수, 메서드 및 클래스 정의에는 파라미터와 반환 타입을 포함한 완전하고 정확한 타입 힌트가 누락 없이 작성되어야 합니다. 개발자의 명시적 요청이 없는 한 `# type: ignore`나 강제 형변환(Type-casting)을 사용하지 마십시오.
- **코드 문서화**: 모든 소스 파일 상단에 파일 수준의 docstring을 누락 없이 기입하고, public 함수 및 클래스에도 명확하고 간결한 docstring을 작성하십시오. 주석은 코드 동작 자체("What")가 아닌 설계 배경이나 결정 이유("Why")에 집중해야 합니다.
- **원자적 변경 (Atomic Changes)**: 변경 사항은 항상 작고 집중되어야 하며 사용자의 직접적인 요청에 부합해야 합니다. 임의로 관련 없는 주변 코드를 정리하거나 대규모 리팩터링을 수행하지 마십시오.
- **안티 패턴**: 
  - 모듈 및 패키지 임포트 시 경로를 추측하여 작성하지 마십시오. 항상 대상 파일의 정확한 위치를 검증 후 임포트 경로를 설정하십시오.
  - 외부 검증 라이브러리(예: Pydantic)를 임의로 추가하여 내부 데이터 검증에 적용하지 마십시오.

## 👨💻 Developer Context
- **Phase 3 범위 제한 준수**: 현재 단계는 Phase 3(로컬 LLM 리뷰 품질 개선 및 실제 프로젝트 대상 분석 안정성 검증)입니다. 범위를 벗어나는 파인튜닝, remediation snippet 생성, 파일 전체 patched code 생성, Python 외 언어 지원, 멀티모달 입력 및 동적 분석 등의 태스크는 즉시 구현하지 않고 README의 `Phase Next`로 이관합니다.

- **협업 거버넌스**: `main` 브랜치에 직접 푸시(Push)하지 않으며, 모든 작업은 GitHub Issue 생성 및 PR 프로세스를 거쳐 Repository Owner의 최종 승인을 받은 후 머지합니다.
