# AI_RULES.md - AI 코드 생성 가이드라인

이 문서는 이 저장소에서 코드를 생성하거나 수정할 때 모든 AI 에이전트(Gemini, Cursor, Copilot 등)가 **반드시** 따라야 하는 핵심 아키텍처, 코딩 표준 및 프로젝트 구조 규칙을 정의합니다.

---

## 1. 기술 스택 및 도구 체인

| 범주 | 도구 및 가이드라인 |
| --- | --- |
| **언어** | Python 3.12+ (최신 Python 기능 사용, `from __future__ import annotations` 포함) |
| **패키지 관리자** | `uv` (`pyproject.toml`을 통해 의존성 관리) |
| **린터 및 포매터** | `ruff` (항상 ruff 포매팅을 준수하며, 기존 규칙을 위반하지 말 것) |
| **테스트** | `pytest` (`tests/` 디렉토리에 위치) |

---

## 2. 코딩 컨벤션 및 스타일

### 2.1. 타입 힌팅 (엄격함)

- **모든** 함수, 메서드, 클래스는 완전하고 정확한 타입 힌트를 가져야 합니다.
- 절대적으로 피할 수 없거나 인간 개발자가 명시적으로 요청한 경우가 아니라면 `# type: ignore`나 명시적인 타입 캐스팅을 사용하지 마십시오.
- 반환 타입과 매개변수 타입은 필수입니다.

### 2.2. 데이터 모델

- 내부 데이터 구조의 경우 Pydantic과 같은 무거운 외부 유효성 검사 라이브러리 대신 `@dataclass(slots=True)`를 사용한 `dataclasses`를 사용하십시오 (`project_nurilab/schemas.py` 참조).
- 직렬화 로직은 명시적으로 유지하십시오 (예: `to_dict()` 구현).

### 2.3. 문서화

- 모든 새로운 Python 파일의 맨 위에는 **파일 수준의 독스트링(docstring)**을 제공하십시오.
- 모든 클래스와 공개 함수에는 간단하고 설명적인 독스트링을 사용하십시오.
- 주석은 "무엇"이 아니라 "왜"에 초점을 맞추십시오 (코드는 그 자체로 설명이 되어야 합니다).

### 2.4. 객체 지향 프로그래밍 (OOP) 및 아키텍처

- 확립된 관심사의 분리(Separation of Concerns)를 따르십시오. 예를 들어, AST 파싱에서 분석기 로직을 AST Visitor 로직과 분리하십시오 (예: `PythonStaticAnalyzer` 및 `_PythonSignalVisitor`).
- 분석기를 확장할 때는 적절한 기본 인터페이스를 상속받고 이를 `project_nurilab/analyzers/`에 배치하십시오.

## 3. 프로젝트 구조 규칙

| 디렉토리 / 파일 | 규칙 / 책임 |
| --- | --- |
| `project_nurilab/` | 메인 애플리케이션 코드. |
| `project_nurilab/schemas.py` | 모든 새로운 데이터 모델과 타입 정의를 여기에 추가하십시오. |
| `project_nurilab/analyzers/` | 새로운 정적 분석 또는 AST 관련 로직을 여기에 배치하십시오. |
| `project_nurilab/llm/` | LLM 상호 작용 및 외부 API 검토와 관련된 로직을 여기에 배치하십시오. |
| `tests/` | 모든 새로운 기능이나 버그 수정에는 해당하는 `pytest` 파일이 동반되어야 합니다. 테스트 파일은 `test_`로 시작해야 합니다 (예: `test_pipeline.py`). |
| `data/` & `tests/fixtures/` | 샘플 코드와 테스트 입력을 여기에 저장하십시오. 루트 디렉토리를 어지럽히지 마십시오. |

---

## 4. AI 에이전트를 위한 실행 지침

- **임포트(Import)를 추측하지 마십시오**: 함수나 클래스를 임포트하기 전에 그 위치를 확인하십시오.
- **테스트 실행**: 핵심 로직을 변경할 때마다 사용자에게 `pytest`를 실행하도록 상기시키거나, 기능이 있는 경우 자율적으로 `pytest`를 실행하십시오.
- **린팅**: 코드 생성 후 출력 결과가 `ruff check .` 및 `ruff format .`을 준수하는지 확인하십시오.
- **원자적 변경 (Atomic Changes)**: 변경 사항을 최소화하고 사용자의 즉각적인 요청에 집중하십시오. 지시가 없는 한 관련 없는 코드를 임의로 "정리"하지 마십시오.
